# ADR 0007: 보안 및 요청 데이터 검증 레이어 도입

## 1. Context (도입 배경 및 문제 상황)

SingleNotion 서비스는 외부 사용자로부터 URL을 입력받아 브라우저 리소스(Puppeteer)를 사용하는 고비용 작업을 수행합니다. 다음과 같은 잠재적 위협에 노출되어 있었습니다:

-  **무분별한 요청**: 악의적인 사용자가 짧은 시간 내에 대량의 변환 요청을 보내 서버 리소스를 고갈시킬 수 있음.

- **잘못된 데이터 주입**: URL 형식이 아니거나 서버가 허용하지 않는 범위의 파라미터(예: 극단적인 pageWidth)를 보내 시스템 에러를 유도함.

- **보안 취약점**: HTTP 응답 헤더를 통해 서버 정보가 노출되거나 XSS 공격 등의 위험이 있음.

## 2. Decision (결정 사항)

시스템의 안정성과 보안을 강화하기 위해 다음 라이브러리 및 정책을 도입하기로 결정했습니다.

1. **처리율 제한 (Rate Limiting)**:  `express-rate-limit`을 사용하여 동일 IP에서의 요청 횟수를 15분당 10회로 제한합니다.

1. **보안 헤더 설정**:  `helmet`  미들웨어를 사용하여 기본적인 보안 헤더를 적용합니다. (CSP의 경우 PDF 미리보기 기능 등을 위해 일부 완화하여 적용)

1. **스키마 기반 검증**:  `joi` 를 사용하여 API로 유입되는 모든 데이터의 타입을 강제하고, 허용된 범위 내의 값만 수용합니다. (예:  `pageWidth` 는 300~5000 사이의 숫자만 허용)

1. **교차 출처 정책**:  `cors` 를 적용하여 허용된 도메인에서의 접근만 관리합니다.

## 3. Consequences (결과)

### 장점

- **시스템 안정성**: 유효하지 않은 요청이 워커(Worker) 단계까지 도달하지 않고 라우터 레벨에서 차단되어 불필요한 리소스 소모를 방지합니다.

- **예측 가능한 에러**: 클라이언트는 400(Bad Request) 또는 429(Too Many Requests) 응답을 통해 요청이 거부된 명확한 이유를 알 수 있습니다.

- **보안성 향상**: 서버 정보 노출 최소화 및 기본적인 웹 공격 방어 체계를 갖추게 되었습니다.

### 단점 및 후속 조치

- **개발 오버헤드**: 새로운 API 엔드포인트를 추가할 때마다 반드시 Joi 스키마를 정의해야 하는 번거로움이 있습니다.

- **사용자 경험**: 정당한 사용자도 처리율 제한에 걸릴 수 있으므로, 에러 메시지를 통해 제한 정책을 명확히 안내해야 합니다.
