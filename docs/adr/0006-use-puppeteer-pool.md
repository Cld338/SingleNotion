# ADR 0006: Puppeteer 브라우저 인스턴스 풀링 도입

## 1. Context (도입 배경 및 문제 상황)
기존 아키텍처에서는 `pdfService.js` 내부에 단일 Puppeteer 브라우저 인스턴스를 유지하고, 수동 배열 큐(`this.queue`)를 통해 동시성을 제어했습니다. 이 방식은 노션과 같이 무거운 페이지 렌더링 시 특정 컨텍스트에서 메모리 누수나 크래시가 발생할 경우, 단일 브라우저 프로세스 전체가 영향을 받아 진행 중인 다른 워커 작업들까지 연쇄적으로 실패하게 만드는 원인이 됩니다.

## 2. Decision (결정 사항)
- 수동 큐 관리 로직을 제거하고, `generic-pool` 라이브러리를 도입하여 브라우저 인스턴스 풀(Pool)을 구축합니다.
- `src/utils/browserPool.js`를 신설하여 인스턴스의 생성, 대여, 반납, 소멸 등의 라이프사이클 관리 책임을 분리(SRP)합니다.
- 환경 변수 `WORKER_CONCURRENCY`를 워커 처리량과 풀의 `Max` 설정 값에 동일하게 적용하여 병목 현상을 최소화합니다.

## 3. Consequences (결과, 장단점 및 후속 조치)
- **장점**: 각 작업이 격리된 브라우저 인스턴스 또는 컨텍스트를 안정적으로 할당받아 특정 작업 실패가 다른 작업에 미치는 영향을 차단합니다. 워커 리소스 사용량 최적화 및 OOM(Out of Memory) 방지에 기여합니다.
- **단점**: `generic-pool`이라는 외부 의존성이 추가됩니다.
- **후속 조치**: `package.json`에 `generic-pool` 의존성을 명시해야 하며, 회귀 테스트를 통해 변경된 자원 반납 로직(`finally` 및 `nodeStream.on('close')`)이 완벽히 작동하는지 검증하는 단위 테스트 코드를 작성해야 합니다.
